The Decision-Making Process (Malczewski, 1999)

  1. Intelligence Phase: What is the problem? What are the opportunities for change? Who are the stakeholders?
  2. Design Phase: What are the alternatives? Define decision criteria and get input data. Develop evaluation model (decision rules).
  3. Choice Phase: Evaluate alternatives (which is best?). Sensitivity analysis. Formulate recommendations.

MCDA (Multi-Criteria Decision Analysis) â€” Key Concepts

Criteria TypesCost criteria: negative impact when values increase (e.g. price, distance to station)
Benefit criteria: positive impact when values increase (e.g. property size)
Constraints: exclude certain alternatives (e.g. nature reserves)
MCDA in 5 Steps1. Know finite set of alternatives
2. Name criteria (cost vs benefit)
3. Standardize criteria (common scale)
4. Assign weights to criteria
5. Aggregate weighted criteria, rank
Standardization MethodsClassification into k classes Â· Linear scale [0..1] (max score, score range) Â· Z-transformation: z=(x-mean)/std (for normally distributed ratio data)
Selecting Criteria (how)1. Literature studies
2. Analytical studies (observation)
3. Survey of opinions (e.g. Delphi method, interviews)

Chapter 2 Â· Criterion Weighting Methods

Method 1: Ranking and Rating

Rank or rate criteria according to preferences, then derive normalized weights. âœ… Simple & very common âŒ No sound basis Â· Not applicable for many criteria (7Â±2 limit)

Method 2: Pairwise Comparison

Criteria are compared pair-by-pair using a matrix. Score: 1 = equally important, up to 9 = extremely more important. Scores normalized by column sums; normalized scores averaged per criterion â†’ weights.

âœ… Sound basis âœ… Allows consistency checks âŒ Pairs grow with nÂ² â€” not scalable to many criteria

Method 3: Analytical Hierarchy Process (AHP)

Solution to the pairwise comparison scalability problem: Define sub-goals, assign criteria to sub-goals, do pairwise comparison within each subgroup. This keeps the number of comparisons manageable.

AHP can also be used as a decision rule (not just for weighting).

Two SDSS Design Approaches

Alternative-driven (Multi-Attribute)

  • Alternatives are already known
  • Need to define criteria and threshold values
  • Example: choosing ski resort from 40 known options

Data/Value-driven (Multi-Objective)

  • Criteria data is known, need to identify alternatives
  • Typically raster-based: each cell = one alternative
  • Example: evacuation planning after chemical accident
ğŸ“Œ In raster-based MCDA, each raster cell represents an alternative. Results can be used as cost surfaces for further analysis (e.g. finding optimal locations or paths).

GIS Methods for SDSS

  • Pre-process input data: coordinate & schema transformations, spatial aggregation, generalization, interpolation
  • Calculate criteria: distances, areas, overlay, map algebra, network functions, DEM calculations (slope, viewshed)
  • Apply decision rules: overlay and aggregation
  • Communicate: geovisualization to illustrate criteria and alternatives

Chapter 2 Â· Sensitivity Analysis & ppGIS

Sensitivity Analysis

How sensitive is the best alternative/decision against potential uncertainties or errors? Sources of uncertainty:

  • Wrong or missing criteria
  • Errors in the criteria datasets
  • Errors in the weighting
Method 1: Error PropagationRequires all potential errors to be known and documented. Rarely the case, especially for soft criteria. Errors must be statistically independent (also rarely guaranteed).
Method 2: Monte CarloGenerate many criteria data sets with slight random variations â†’ run the decision model with each â†’ observe which alternatives change â†’ provides overview of sensitivity. May require computing power.

ppGIS (Public Participation GIS)

Techniques to incorporate multiple stakeholders (with conflicting objectives) into the decision-finding process. Overlaps with GDI concepts (e.g., internet map with commenting function). Terms not yet well-defined (Schlossberg & Shuford, 2005).

What is INSPIRE?

INSPIRE = INfrastructure for SPatial InfoRmation in Europe
Started Sept. 2001. INSPIRE Framework Directive adopted April 2007, took effect May 2007. First EU legislation focusing on geoinformation.

Goals of INSPIRE

  • Support integrated European Environmental Policies
  • Build on national GDIs and international standards
  • Use existing digital geodata and spatial data services of the public sector
  • Follows a stepwise implementation plan

5 Core Components of INSPIRE

4. Data and Service SharingEase access for authorities. No fees or minimal fees. No distorting market competition. No access to critical infrastructure / protected species. No interference with IP rights
5. Monitoring & ReportingMember states: establish contact points, adopt directive into national law (by 2010), annual reports. EC: coordinates, reports every 6 years to European Parliament

INSPIRE Rules: Implementing Rules vs. Guidance Documents

Implementing Rules (legally binding)

  • Part of INSPIRE Framework Directive
  • Define mid/long-term requirements ("WHAT")
  • Regulate the minimum required

Guidance Documents (not legally binding)

  • Tell "HOW"
  • Example: metadata conforms to ISO 19115 / ISO 19119 / ISO 15836
  • Can be updated as technologies change

Chapter 4 Â· Interoperability for Geoinformation

Definitions

General (IEEE, 1990): "The ability of two or more systems or components to exchange information and to use the information that has been exchanged."
For GI (INSPIRE, 2007): "The possibility for spatial data sets to be combined, and for services to interact, without repetitive manual intervention, in such a way that the result is coherent and the added value is enhanced."

Requirements for Information System Interoperability

1. InterfacesWell-defined communication interfaces (e.g. HTTP)
2. EncodingWell-defined common language for transferring information (e.g. XML)
3. IdentifiersUnique identifiers to find and allocate objects (e.g. URI, URL)
4. StandardsSpecifications (standards/norms/contracts) describing the above, agreed upon by consensus organizations

ISO 19119 Terminology

ServiceA distinct part of the functionality that is provided by an entity through interfaces (e.g. "WMS")
InterfaceA named set of operations that characterize the behavior of an entity (e.g. "GetMap interface")
OperationA specification of a transformation or query that an object may be called to execute. Has a name and list of parameters (e.g. "GetMap(bbox, layers, â€¦)")

History of GI Interoperability

~1985Data-Exchange-Format Era: GIS on desktop PCs, proprietary monolithic systems, data exchange via files, no open APIs
~1995GI-Interoperability-via-API Era: GIS get distributed (multi-layer architectures), GeoMiddleware (ArcSDE, Oracle-Spatial), move towards APIs â€” but still proprietary. Metadata standardization becomes hot topic
