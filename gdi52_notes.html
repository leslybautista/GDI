Integration ‚Äî Transformation of Geodata into Common Models

Schema Mapping ¬∑ Schema Translation ¬∑ Complexity Levels ¬∑ Tools

Types of Transformations

Coordinate Transformations

Converting geometries between different CRS (e.g., Gau√ü-Kr√ºger ‚Üí ETRS89/UTM). Purely geometric ‚Äî does not change the data model or attribute meaning.

Schema Transformations

Converting/transferring between different conceptual data schemata. Addresses structure and meaning, not just position. Three aspects:

Syntactic aspect

Change of data types or formats. Example: double ‚Üí real, ASCII ‚Üí Binary, CSV ‚Üí GML encoding.

Schematic aspect

Change of data structure or conceptual model. Example: object-oriented ‚Üí relational model, vector ‚Üí raster.

Semantic aspect

Transformation of content/meaning. Example: ratio-scaled ‚Üí ordinal-scaled, ATKIS land-use classes ‚Üí INSPIRE land-use classes.

Schema Mapping vs. Schema Translation

Key distinction: Schema Mapping = defining the rules (intellectual work, mostly still manual). Schema Translation = applying those rules to actually transform the data (automated). Both steps are always needed.

1. Schema Mapping

What: Deriving and defining the transformation rules, usually at the conceptual level ‚Äî comparing UML diagrams and textual descriptions of feature types.

Schema Matching = the (partly automated) search for correspondences between different schemata. A sub-step of mapping.

Preprocessing is still mostly manual. Tools support formalizing the mapping rules (e.g., FME, HALE).

2. Schema Translation

What: The automated data transformation from source schema to target schema, applying the defined mapping rules.

Tools: FME (Feature Manipulation Engine), HALE (Humboldt Alignment Editor, open source by WeTransform), Snowflake GO Publisher, ArcGIS for INSPIRE.

Can be batch (one-time), or demand-driven/ad-hoc (triggered by a query in real-time).

  • Reduction: Deriving INSPIRE Cadastral Parcels from ALKIS Flurst√ºcke, excluding all ownership attributes. (Source has more attributes than target ‚Äî you throw some away.)
  • Fusion: INSPIRE Transportation Networks enriched with data from Stra√üeninformationsbank (TT-SIB) + municipalities + public transport companies. (Multiple sources ‚Üí one target dataset.)
  • Selection: Using TT-SIB as source for INSPIRE higher-level road network rather than ATKIS, because TT-SIB covers motorways more comprehensively. (Choosing the best-fit source.)

Demand-Driven Schema Translation

Rather than a one-time batch translation, it is possible to translate data on-demand: a user queries for INSPIRE data ‚Üí the translation service queries the source (ATKIS) ‚Üí returns translated results in real-time. Requirements: (1) fully formalized Schema Mapping, (2) automated Schema Translation with sufficient performance.

Workflow: INSPIRE Data Provision in Saxony

2

3
Define schema transformations ‚Äî specify schema mapping rules using UML model comparison.
4
Translate ‚Äî run the automated translation using FME/HALE/etc.
5
Validate ‚Äî use the INSPIRE Validator (inspire.ec.europa.eu/validator/) to confirm conformance with the target schema.

Integration ‚Äî Fusion of Heterogeneous Geodata

Object Matching ¬∑ Similarity Measures ¬∑ Service-Based Fusion ¬∑ GDI Workflow

What is Geodata Fusion?

Yuan & Tao (1999) ‚Äî Conflation: "Combining information from two or more geo-datasets to make a master dataset that is superior to either source in either spatial or attribute aspect."
Bleiholder & Naumann (2008): "Fusing multiple records representing the same real-world object into a single, consistent, and clean representation."
OWS Fusion Study (2009): "Combining or associating data/information regarding one or more entities to improve one's capability for detection, identification, or characterization of that entity."

Levels and Classification of Geodata Fusion

Fusion LevelWhat is FusedExample
Signal / Sensor FusionRaw sensor signalsCombining multispectral satellite bands into a false-color composite
Data FusionData recordsCombining GPS tracking data with loop detector counts for traffic analysis
Information FusionInterpreted informationFusing different news channels to build a comprehensive event picture

Horizontal Fusion

Joining datasets covering different geographic areas (no overlap). Challenge: boundary consistency ‚Äî objects at the border may be inconsistently split or labeled across the boundary.

Vertical Fusion (focus of this lecture)

Joining datasets covering the same area with different attributes or levels of detail. Subtypes:

  • Updating: replace outdated data with newer data
  • Change Detection: identify what has changed between two time stamps
  • Information Enrichment: add attributes from one source to features from another
  • Data Integration: full merge into a superior, unified dataset

General Fusion Workflow

1
Input Data ‚Äî two or more source datasets covering the same phenomena.
2
Pre-processing ‚Äî subsetting, generalization, coordinate transformation to a common CRS. This is equivalent to the schema mapping/matching step. Covers Schema, Geometry, Topology, and Semantics aspects.
3
Matching ‚Äî identify homologous objects: which feature in dataset A represents the same real-world object as which feature in dataset B? Core intellectual challenge.
4
Fusion ‚Äî merge matched objects; resolve attribute conflicts; describe uncertainty about unmatched objects.
5
Post-processing ‚Äî quality checks, consistency validation, final formatting.
6
Result ‚Äî the fused, improved dataset.

GDI Fusion Workflow (with Metadata)

In a GDI context, the workflow is extended with metadata-driven steps: Discovery & Evaluation (find fusion candidates via catalogue/metadata) ‚Üí Compare & Harmonise (coordinate transformations, schema alignment, generalisation) ‚Üí Conflict Analysis (duplicate detection, change detection) ‚Üí Matching (similarity measurement, link identification) ‚Üí Data Fusion (merge, resolve conflicts, document uncertainty).

Object Matching Strategies

Object (feature) matching = finding homologous objects ‚Äî features in different datasets that describe the same real-world phenomenon (e.g., the same road in ATKIS and OSM).

Bottom-Up (Geometry-first)

Common points ‚Üí common edges ‚Üí then validate with semantics. Best when geometry is reliable and precise.

Top-Down (Semantics-first)

Common semantic classes (e.g., "all local streets") ‚Üí common edges ‚Üí common points. Best when class definitions are clear.

In practice: both strategies are combined using a composite similarity score from three dimensions:

Similarity Measures

DimensionMeasures
GeometryPosition, length, area, perimeter, sinuosity; percentage area overlap (optionally with tolerance buffers); Hausdorff distance = max. of minimal distances between two point sets (how far apart at worst case); Fr√©chet distance = measures similarity between two curves by considering the order of points (like a leash between a person and dog each walking one path)
TopologyRound-Trip-Walk: compare the sequence of neighboring edges and nodes ‚Äî if two road networks are homologous, their connectivity graph should be similar
SemanticsNumerical differences on attribute values; Damerau-Levenshtein distance between road names (counts character insertions, deletions, substitutions, transpositions); comparing XML tree structures; reasoning with thesauri/ontologies to bridge classification differences

Common Matching Problems

Different Representations

The same road: single centerline in ATKIS vs. two parallel lines in a detailed cadastral dataset. Cannot be matched geometrically without understanding this representational difference.

Different Segmentations

Same continuous road split at different points in the two datasets ‚Äî ATKIS splits at admin boundaries, OSM splits at intersections. Segment lengths and IDs are different.

Missing Objects

A small private road exists in OSM but not in ATKIS (ATKIS only captures roads of certain importance). No homologue to match to ‚Üí "missing object" problem.

Different Classifications

The same road: "Hauptstra√üe" in ATKIS, "secondary" in OSM. Semantic mapping needed across different classification systems.

Case Study: ATKIS + OSM Ad-Hoc Fusion (Wiemann & Bernard 2010)

  • Goal: Update and enrich ATKIS road data with VGI from OSM; test feasibility of service-based ad-hoc fusion in a GDI
  • Service Architecture: Two WFS (ATKIS source + OSM source) ‚Üí WPS (fusion processing) ‚Üí WMS (visualization of fused results)
  • Matching method: Additive confidence measure combining geometric + semantic similarity scores. Example weights:
Confidence measure for ATKIS/OSM road matching: +0.1 if bounding boxes intersect or distance < 50m ¬∑ +0.2 if angle difference < œÄ/10 ¬∑ +0.1 if minimum geographic distance < 50m ¬∑ +0.1 if length difference < 20m ¬∑ +0.15 if Hausdorff distance < 50m ¬∑ +0.15 if topology relation is not disjoint ¬∑ +0.2 if Damerau-Levenshtein distance between road names < 4

Advanced: Linked Data for Fusion Relations (Wiemann & Bernard 2015)

Fusion relationships between homologous objects are stored persistently in a Triple Store (Linked Data / RDF). SPARQL queries then efficiently retrieve pre-computed matches. A UML data model formalizes the relations: FeatureRelation (with timestamp) links 2+ FeatureResource objects, associated with RelationType (is-same, sub-set, super-set‚Ä¶), SimilarityMeasure, and ConfidenceMeasure.

Advantage: Avoids recomputing matches on every request. Supports incremental updates. Enables reasoning over the fusion network.

Other Fusion Examples

Sensor + Satellite Data Fusion: In-situ air quality measurements (point-based, high temporal resolution, quality-assured) fused with satellite images (spatial coverage, coarse resolution) using spatial interpolation methods such as IDW + multiple regression, or Co-Kriging. Result: spatially complete air quality maps with high temporal frequency.

Conclusion ‚Äî Geodata Integration

Geodata Transformation: Translating between conceptual models ‚Äî Schema Mapping (define rules, still mostly manual) + Schema Translation (apply rules automatically, tools: FME/HALE). Demand-driven translation is technically possible but requires fully formalized mapping.

Geodata Fusion: Combining datasets to produce something better. Core challenge = object matching using geometry, topology, and semantic similarity. Schema transformation is often a prerequisite step in fusion. Role of metadata: helps discover fusion candidates, provides quality info, helps resolve semantic ambiguity.

Discovery ‚Äî Metadata for Geodata and GI Services

ISO 19115 ¬∑ Dublin Core ¬∑ DCAT ¬∑ Thesauri ¬∑ GEMET

Why Metadata and Discovery Services?

INSPIRE requirements for discovery services: ability to find geodata and GI services by keywords, classifications (taxonomies), data quality, conformity with standards, spatial and temporal coverage, access/license conditions, responsible institutions. Ideally also: usability and fitness-for-purpose indicators.

Three Roles of Metadata

1. Discovery

Spatial/temporal coverage, keywords on topics, responsible institution. Helps users find datasets or services that might be relevant.

2. Evaluation of Usability

Geometry quality, thematic quality, scale/resolution, lineage. Helps users decide if a dataset is fit for their specific purpose.

3. Integration Support

Syntactic (formats, protocols, system environments), Semantic (CRS, classifications, vocabularies/thesauri, ontologies), and Legal (licensing, access conditions) aspects.

Two Strategies for Metadata Organization

Non-Structured Metadata (Google-style)

  • No standardized common schema; metadata not made explicit
  • Search engines auto-generate internal metadata from indexed web pages
  • ‚úÖ Fully automated acquisition
  • ‚ùå Variable quality and completeness
  • ‚ùå Only finds online, crawlable resources

Structured Metadata (Catalogue-based)

  • Follows standardized schema; metadata made explicit and published
  • Catalogue applications allow structured discovery
  • Traditionally from library practice; standard approach for geodata
  • ‚úÖ Homogeneous quality (ideally); also covers non-online resources
  • ‚ùå Generation/maintenance is mostly semi-automatic; needs human input

ISO 19115 ‚Äî Service Coupling Types (formerly ISO 19119)

ISO 19115-1:2014 now covers metadata for GI services (previously in ISO 19119). Three coupling types between a service and its data:

  • Tightly coupled: service is permanently associated with a specific dataset instance. Metadata must describe both service AND the coupled data (using ISO 19115).
  • Loosely coupled: service is not associated with specific data. Metadata describes the service interface only.
  • Mixed: combination.

The "Chicken and Egg" Problem of Geo-Metadata

Data producers are typically reluctant to create, provide, and maintain good quality metadata. Without good metadata, users cannot find/evaluate data ‚Üí the benefit of metadata is not experienced ‚Üí no motivation to improve it. Circular. Today's GIS applications do not fully support automatic metadata generation (e.g., logging processing lineage). Open research challenges: better automatic metadata acquisition; incorporating user ratings; using lineage metadata to check for meaningful geoprocessing.

Example: Meaningful Geoprocessing Using Metadata

A practical example of metadata used to prevent errors: if metadata says a dataset is nominal/ordinal scaled (land use classes), then a geoprocessing tool should refuse to apply bilinear interpolation (which assumes ratio-scale data) and use nearest-neighbor instead. This check can be automated if the scale-level is stored in metadata.

Thesauri for Semantic Interoperability

Thesaurus: A form of controlled vocabulary that minimizes semantic ambiguity by ensuring uniformity and consistency in the storage and retrieval of content objects. Provides: preferred terms, synonyms, broader terms, narrower terms, related terms, and definitions.

Uses in GDI:

  • Generate standardized keywords as metadata entries ‚Üí all data custodians use the same controlled terms
  • Support searches by also searching for synonyms or broader/narrower terms ‚Üí searching "water quality" also finds datasets tagged with "nitrate", "dissolved oxygen" (narrower terms)
GEMET ‚Äî GEneral Multilingual Environmental Thesaurus: Developed by the European Environment Agency for environmental data across Europe. Contains all INSPIRE Spatial Data Themes ‚Üí used for harmonizing INSPIRE metadata keywords. Multilingual (all EU official languages). Each entry has a persistent URL, human-readable definition, broader/narrower/related terms, and links to other thesauri and descriptions. Available at: eionet.europa.eu/gemet.
Section 5.5.2

Catalogue Services ‚Äî CSW & OGC API Records

OpenGIS CSW ¬∑ CSW-T ¬∑ INSPIRE Discovery ¬∑ OGC API Records

Catalogue Services Web (CSW)

CSW (OpenGIS Catalogue Service Web): OGC specification for interoperable geo-catalogue services. Current versions: 3.0 (2016) and 2.0.2 (2007). Protocol: HTTP GET/POST (and SOAP). Stateless. Follows the Publish-Find-Bind paradigm: data providers publish metadata, users find and evaluate resources, services are discovered and bound.

Two Types of CSW

CSW (Read-Only)

For discovery ‚Äî requesting and querying metadata records. Does not support writing metadata. Operations: GetCapabilities, DescribeRecord, GetDomain, GetRecords, GetRecordById.

CSW-T (Transactional)

Adds write operations for storing metadata. Additional operations: Transaction (push) and Harvest (pull). Can insert, update, and delete metadata records.

CSW Operations in Detail

GetCapabilities
Mandatory Returns metadata about the CSW instance itself: name, provider, which operations are supported, which metadata schema is used, query capabilities. Analogous to WMS/WFS GetCapabilities.
DescribeRecord
Mandatory Returns a description of the metadata schema used by this catalogue. Analogous to WFS DescribeFeatureType ‚Äî tells clients which attributes are available for querying and their data types.
GetDomain
Optional Provides information about the domains (min/max values, available discrete options) of specific metadata attributes ‚Äî helps clients formulate meaningful queries (e.g., "which keywords are actually available?").
GetRecords
Mandatory The main search operation. Analogous to WFS GetFeature ‚Äî query metadata using offered criteria. Response = list of matching metadata records. Can request sorted results. Can trigger a distributed search across linked catalogues.
GetRecordById
Mandatory Retrieve a specific metadata record by its unique ID. Used to display full metadata details after a user selects a result from GetRecords.
Transaction
CSW-T only Push mechanism ‚Äî allows data custodians to insert, update, or delete metadata records in the catalogue database. Analogous to WFS Transaction.
Harvest
CSW-T only Pull mechanism ‚Äî triggers the catalogue to automatically read/fetch metadata from other metadata holdings (by URI). Can be ad-hoc (synchronous) or scheduled at regular intervals (asynchronous, e.g., nightly). Harvesting solves distributed search performance issues by creating local copies.

Distributed Search vs. Harvesting

Distributed Search

A query is forwarded in real-time to multiple linked catalogue servers. Results are aggregated. Problems: slow if remote servers are unavailable or slow; overall response time depends on slowest server.

Harvesting (Solution)

Metadata from remote catalogues is pre-fetched and stored locally at regular intervals. Local queries run on locally cached data ‚Üí fast, reliable. Tradeoff: slight lag ‚Äî local data may be up to one harvesting interval old.

Metadata Publishing in GDI Saxony (4 Channels)

  1. Via a web editor ‚Äî manual entry by data custodians
  2. Import from XML files ‚Äî batch import of ISO 19139 XML documents
  3. Harvesting via CSW interfaces ‚Äî automatic pull from other CSW servers
  4. Harvesting from an XML web folder ‚Äî automatic pull from file directories on web servers

CSW ISO Profile

Specifies how to use CSW with ISO 19115 metadata. Distinguishes resource types: Dataset, Dataset collection, Service (tightly/loosely/mixed coupled), and Application (web apps like a WMS client). This is the profile recommended by INSPIRE for discovery service implementations.

INSPIRE Discovery Service

INSPIRE defines a Discovery Service as a catalogue service for publication, search, evaluation, and usage of INSPIRE resources using metadata conforming to the INSPIRE Metadata Implementing Rule (ISO 19115 INSPIRE profile).

INSPIRE Discovery Service QoS (legally binding): Performance = first records returned within 3 seconds. Capacity = 30 parallel queries/second. Availability = 99% (‚â§4 days/year downtime).

CSW Discussion ‚Äî Limitations

  • Service metadata can link to dataset metadata, but this is mostly unidirectional ‚Äî it's hard to link from a dataset back to the services that can serve it
  • Lineage/provenance metadata (the processing history of a dataset) is hardly supported in current CSW implementations
  • Distributed search performance issues exist ‚Üí harvesting is the practical solution
  • Emerging Open Data portals use DCAT/GeoDCAT-AP ‚Äî increasing convergence with traditional ISO/CSW approaches

A multi-part RESTful API specification for creating, modifying, and querying metadata records. The modern equivalent of CSW.

Geoportals

Requirements ¬∑ Gazetteers ¬∑ Access Control ¬∑ Usability ¬∑ Examples

What is a Geoportal?

A geoportal is a GDI application that integrates and groups the distributed information offerings within a GDI, providing a web-based user-facing entry point to geodata, services, and related applications. Geoportals may be directed to specific user groups and may offer APIs or tools for developing custom applications. They also reflect ‚Äî and demonstrate ‚Äî the development status of a GDI.

Improving geoportal usability is receiving increasing attention. A geoportal's perceived quality directly depends on the quality of the underlying GDI services and data ‚Äî a portal cannot overcome bad underlying QoS.

Typical Geoportal Functional Requirements

Discovery & Search

  • Metadata search via Web Catalogue Services (CSW)
  • Place-based search using geographic names (Gazetteers)
  • Keyword-based search using thesauri (e.g., GEMET)
  • Interactive exploration of discovered metadata (preview via WMS)

Service Combination & Processing

  • Guided access to distributed GI services
  • Support for combining/coupling services (e.g., WPS for coordinate transformations, reclassifications)
  • Note: truly guided service composition is hardly operational today
  • Positive exception: US GeoPlatform (geoplatform.gov) ‚Äî a suite of tools for integrating and analyzing data

Access Control (AAA)

  • Authentication: verify user identity (who are you?)
  • Authorization: control access rights (what are you allowed?)
  • Accounting: usage tracking, billing, ordering
  • User profiles and personalized offerings
  • Geo-specific OGC specs exist (GeoXACML, OGC Digital Rights Management‚Ä¶) but have almost no practical uptake ‚Äî general IT security standards are used instead

Additional Resources & Multilingualism

  • Links to web applications using the provided data/services
  • Documents, FAQs, links to other geoportals, legal information
  • Multilingual interfaces (mandatory for INSPIRE Geoportal across all EU languages)
  • Multilingual thesauri and gazetteers for cross-border search

Gazetteers ‚Äî Place-Based Search

Gazetteer: A digital geographical dictionary that links geographic names to geometries (and vice versa). Described by ISO 19112. Allows: name ‚Üí coordinates (geocoding) and coordinates ‚Üí name(s) (reverse geocoding).

Characteristics of gazetteers:

  • Traditionally link to point geometries; modern gazetteers also link to lines and polygons
  • Hierarchical organization of names ‚Üí allows disambiguation (Dublin, Ireland vs. Dublin, Ohio, USA)
  • May offer alternative, multilingual, or historical names (Chemnitz ‚Üî Karl-Marx-Stadt; K√∂ln ‚Üî Cologne)
  • Feature type classification (city, river, mountain, administrative unit)
  • Links are not necessarily unique ‚Äî one name may correspond to multiple places

Examples:

  • Geocoding: "Helmholtzstra√üe 10, 01069 Dresden" ‚Üí UTM point
  • Line lookup: "Elbe (River)" ‚Üí polyline in Gau√ü-Kr√ºger
  • Reverse geocoding: polygon 90% overlapping Saxony ‚Üí "Sachsen, Deutschland"

Interoperability: OpenGIS Gazetteer WFS Profile (WFS-G) provides a standard API for gazetteers. Real examples: GeoNames (geonames.org, global, free, WGS84), NGA GEOnet Names Server (US DoD), BKG GeocorderPlus (Germany).

INSPIRE Geoportal Architecture

The EU INSPIRE Geoportal aggregates services from all EU member states. Components: Thesauri & Translation service ‚Üî Catalogues (distributed CSW) ‚Üí INSPIRE Geoportal UI ‚Üê Gazetteers (WFS). The portal accesses WFS (data), WMS (visualization), WCS (coverage data), and WCatS (catalogue search) from various national providers. Multilingualism is built in.

Geoportal Usability

Two key dimensions:

  1. Functionality, performance, reliability ‚Äî depends on underlying services and data quality. INSPIRE QoS requirements (WMS: 5s/470KB, 20 parallel requests/s, 99% availability) set minimum expectations. Users notice poor QoS immediately.
  2. User interface design ‚Äî studied through user surveys and expert evaluations. Key finding from Geoportal Sachsen usability study: users strongly prefer a single, central search field over separate spatial and thematic search fields. Also: providing context about accuracy of coordinate calculations significantly improves information usability.

Implementation Strategies ‚Äî Reference Models (RM-ODP)

What is a Reference Model ¬∑ 5 ODP Viewpoints ¬∑ GI Service Chains ¬∑ Worked Scenario

What is a Reference Model?

OASIS: "An abstract framework for understanding significant relationships among entities of some environment, for developing consistent standards. Not tied to specific technologies or implementations ‚Äî provides common semantics across different implementations."

In the GDI context, reference models support the design, implementation, and documentation of distributed information systems by providing a structured framework and best practice examples. They are abstract enough to be technology-independent, but concrete enough to guide actual architecture choices.

RM-ODP ‚Äî Reference Model of Open Distributed Processing

ISO/IEC 10746 ‚Äî RM-ODP: The most prominent reference model for GDI implementations. Defines 5 complementary viewpoints for describing a distributed system. Each viewpoint focuses on a different set of concerns and stakeholders, using different levels of abstraction. All 5 viewpoints together provide a complete description of the system.

Note: An earlier, related framework ‚Äî the "4+1 View Model" (Kruchten 1995) ‚Äî uses 4 views (Logical, Process, Development, Physical) plus a central Scenarios view. The ODP model extends and formalizes this thinking.

The 5 RM-ODP Viewpoints

1. Enterprise / Business
Management and user/usage perspective. Describes policies, roles, responsibilities, and high-level business processes. Answers: Who are the stakeholders? What are the organizational rules? How should data ownership and service operation be arranged ‚Äî centrally or decentrally? Example for INSPIRE: three valid options ‚Äî central operation (one authority runs everything), decentral operation (each data holder runs their own services), or mixed.
2. Information
Common data structures and semantics. Describes the data/information model ‚Äî what data is exchanged and what does it mean? Based on ISO 19109 General Feature Model. Examples: AAA application schema (ALKIS/ATKIS definitions), INSPIRE Data Specifications (UML models of INSPIRE features). This is the "what does the data look like" viewpoint.
3. Computational
Abstract service interfaces and functions. Defines which service categories are needed and how they interact ‚Äî without specifying technology. Based on ISO 19119 service categories. Also defines types of GI service chains. This is the "what services are needed and how do they communicate" viewpoint.
4. Engineering / Technology
Concrete technology and architecture. Specifies which OGC/ISO standards are used for each service component, interface specifications, collaboration and sequence diagrams. Also includes non-functional requirements (QoS, security). Example: for a routing system ‚Üí WMS for visualization, WFS for road data, WPS for routing computation, GML for encoding, CSW for service discovery.
Cross-cutting
Scenarios / Use Cases: Cross-cutting tool used throughout all viewpoints. Scenarios ground the design in concrete user tasks, drive the identification of required services (computational viewpoint), and communicate the system to non-technical stakeholders (enterprise viewpoint).

Types of GI Service Chains (Computational Viewpoint)

User Defined Chaining
(transparent)
The user manually searches, finds, and combines services. Full visibility and control. Example: a GIS expert uses GeoMIS Sachsen to find a WMS for flood data, then manually adds it to their GIS client. The user sees and controls every step.
Workflow-Managed Chaining
(translucent)
A Workflow Enactment Service manages execution of a pre-defined service chain. The user defines the workflow, the system executes it. Analogous to ArcGIS Model Builder or a BPEL workflow engine. User sees the workflow structure but not individual service call mechanics.
Aggregate Service
(opaque)
A black-box service hides all internal service composition. User only sees one interface ‚Äî the internal chain of WFS + WPS + WMS etc. is completely hidden. Example: a routing web service that internally calls a road data WFS, a routing WPS, and a visualization WMS, but presents a single "Get Route" interface to the user.

ODP Engineering Viewpoint ‚Äî What it Must Specify

  • Interface specifications for each service component (e.g., WMS 1.3.0 for visualization, WFS 2.0 for data access)
  • Collaboration/sequence diagrams showing how services interact in time (which service calls which in what order)
  • Non-functional requirements: QoS (performance, capacity, availability ‚Äî as defined by INSPIRE), security (HTTPS, authentication), scalability, error handling

Worked Example Scenario: Cross-Border Foot-and-Mouth Disease (NRW, 2003)

A classic pre-INSPIRE cross-border GDI use case scenario studied by TU Dresden / GDI NRW:

Enterprise Viewpoint

A Dutch veterinarian detects foot-and-mouth disease on a farm near the German border. They need to share location data with German colleagues for response planning. Two national GDIs (Dutch NCGI, German GDI NRW) must cooperate while maintaining data sovereignty.

Computational Viewpoint

Required service categories: WMS (visualization), CSW (discovery), distributed catalogue search with cross-language query translation (Dutch ‚Üî German ‚Üî English). Service chain type: User Defined Chaining ‚Äî the German veterinarian manually finds and combines WMS layers from both countries.

Engineering Viewpoint

Step by step: German vet opens CSW client ‚Üí queries GDI NRW CSW for WMS on "Gelderland, Maul-und-Klauen-Seuche" ‚Üí distributed search forwards query to Dutch CSW (NCGI, Amsterdam) ‚Üí Dutch WMS URL returned ‚Üí WMS URL added to map client ‚Üí further WMS layers added (Dortmund, Bonn, Amsterdam) ‚Üí composite cross-border map ready.

Conclusions on Reference Models

  • RM-ODP's 5 viewpoints provide a complete, layered design framework from business requirements to concrete technology choices
  • Typically: Information viewpoint ‚Üí ISO 19xxx feature model; Computational viewpoint ‚Üí SOA design with OGC Web Services
  • Scenarios/use cases are essential for driving and communicating requirements at all viewpoints
  • Iterative user requirement analysis plays a central role throughout the design process
  • The approach is consistent with how INSPIRE's own architecture is documented (INSPIRE Network Services Architecture v3.0)
