5.1 Software Architectures for GDI

Motivation ¬∑ Layered Systems ¬∑ Tight vs Loose Coupling ¬∑ SOA ¬∑ REST/ROA ¬∑ Linked Data ¬∑ OpenAPI

Why Architectures?

Software architectures serve to structure complex systems, communicate design decisions, coordinate development, and enable cost reduction through component reuse and easier maintenance. The architect knows what to do, when, and how ‚Äî analogous to construction blueprints.

Definition (Bass et al. 2003): "The software architecture of a system comprises its software components, the externally visible properties of those components, and the relationships among them."
Origin (Brooks & Iverson, 1969): "Architecture tells what happens; implementation tells how it is made to happen."

Two main goals for distributed systems: platform independence (independence from hardware/software platforms) and reusability (for different tasks and purposes).

Layered Systems

Layering is a fundamental principle of all distributed systems. Each layer encapsulates and abstracts the layer below it, with well-defined interfaces between layers.

2-tier (since 1980s)

Client (UI + process mgmt) ‚Üî Server (DBMS + process mgmt). Classic client/server.

3-tier (since 1990s)

Client (UI) ‚Üî Application Server (business logic, middleware) ‚Üî Data Server (DBMS). Foundation of most today's GIS.

ISO OSI Model

7-layer network model. Example of strict layering: layer n may only access layer n-1.

Java Example: Java Program ‚Üí Java API + Compiler ‚Üí Java Virtual Machine ‚Üí OS (Linux/Windows/...). The JVM is the "layer" that provides platform independence.

Forms of layering: Strictly layered (layer n may only access layer n-1) vs. non-strictly layered (can skip layers). Strictly layered = more disciplined, better encapsulation.

Tight vs. Loose Coupling

Traditional communication was based on sockets or remote procedure calls (RPC) ‚Äî OS-dependent, tedious to implement, non-interoperable. Today, two paradigms:

Service Oriented Architecture (SOA)

Goal: Dynamic system integration ‚Äî coupling application components (services) at runtime. Services offer their functionality through well-defined interfaces via a network.

Publish-Find-Bind Paradigm: (1) Service Provider publishes service description to a Service Registry. (2) Service Requestor finds the service in the registry. (3) Requestor binds to and calls the provider directly.

SOA Technologies

üî§ XML ‚Äî Extensible Markup Language

  • Data in hierarchical text file structure (object tree)
  • XML Schema: describes structure and data types (own XML dialect)
  • XML Namespace: uniquely named elements (e.g., xmlns="http://earth.google.com/kml/2.0")
  • XSLT: transformation rules between XML dialects or to HTML/PDF
  • Prominent dialects: SVG, GML, KML, WSDL
  • Well-formed = conforms to basic XML syntax (properly nested, one root)

üìã WSDL ‚Äî Web Services Description Language

  • XML spec for describing web services (service metadata)
  • Platform-, language-, and protocol-independent
  • Typically auto-generated in development environments
  • Client reads WSDL ‚Üí knows which functions are available
  • Combined with SOAP + XML Schema for full SOA stack

üåê HTTP ‚Äî HyperText Transfer Protocol

  • GET: data + query in URL as Key-Value Pairs (KVPs): KEY=value&KEY2=val2. Limited URL length. '?' separates URL from query, '&' separates KVPs.
  • POST: data sent separately (usually as XML). Handles larger data volumes.

üì¶ SOAP ‚Äî Simple Object Access Protocol

  • Originally acronym (now just "SOAP")
  • Can use any transport protocol, but usually HTTP
  • SOAP message is an XML file: Envelope ‚Üí Header (meta-info for server) + Body (actual request/response) + optional Attachment
  • Sent as HTTP/POST request
  • Currently recommended for INSPIRE services
  • Criticized for being too generic and complex

Registries

Original plan: UDDI (Universal Description, Discovery and Integration) as "yellow pages" for web services ‚Äî stopped in 2006. Today, multiple co-existing approaches:

  • GI specific: OGC Catalogue Services (CSW) + ISO 19115/19 metadata; OGC API ‚Äì Records; STAC (SpatioTemporal Asset Catalogs)
  • Open data: DCAT (Data Catalog Vocabulary); GeoDCAT-AP (for European data portals)

Key insight: SOA and REST/ROA are complementary ‚Äî a SOA-based system can be linked into REST-based systems and vice versa. The differences are not as large as early debates suggested.

OpenAPI

OpenAPI Specification (OAS) is an industry standard for RESTful API design. Originally based on Swagger 2.0 (donated by SmartBear, 2015), maintained by the OpenAPI Initiative (OAI). Current version: OAS v3.1.0 (Feb 2021).

  • HTTP-based, non-proprietary, formally described, machine- and human-readable
  • Described in JSON or YAML format
  • Server and client code can be auto-generated from an OpenAPI Document

An OpenAPI document contains: openapi (version), info (title + version), paths (endpoints with parameters and responses), Operation Objects (summary, parameters, payload, possible responses), Responses Object (HTTP response codes + meanings).

Huge tool support: code generation, validation, documentation, Swagger editor, converters.

OGC Context: Traditional OGC Services (WMS, WFS, WCS) are SOA-based. The new OGC API family (OGC API ‚Äì Maps, Features, Tiles, Styles, Coverages) is ROA/OpenAPI-based. Old services date from ~2000; OGC APIs first became available in 2021.

Why traditional OGC services are not enough (W3C, 2017): Non-expert users can't just "follow links"; queries are complex and require domain knowledge; search engines cannot index OGC Web service content (only catalogue services intended for GDI discovery, not general-purpose web).

Linked Data

Web evolution: Web of documents (1989-200x): HTML pages linked via hyperlinks. ‚Üí Semantic Web & Linked Data (Berners-Lee, 2006): data-oriented, using common ontologies, goal to publish open data that can easily be combined.

Linked Data Principles (Berners-Lee):

  1. Use URIs to identify objects (and use HTTP URIs so they can be looked up)
  2. Data provided using RDF (Resource Description Framework ‚Äî an XML dialect)
  3. Links lead to common vocabularies or other related objects/data (also RDF-based)

RDF Triples

Linked Data describes everything as Subject ‚Äì Predicate ‚Äì Object triples:

Example: Helmholtzstra√üe geo:sfIntersects Georg-B√§hr-Stra√üe (Subject URI) (Predicate/vocab) (Object URI) The geometry is stored as a literal (GML string) attached to each object.

SPARQL: query language for Linked Data (analogous to SQL for relational databases). The Linked Open Data Cloud (May 2020): 1,269 datasets with 16,201 links. Geography nodes include OSM, NUTS, Ordnance Survey Linked Data.

GDI Conclusion: Linked Data is high implementation complexity, low domain dependence (very open). Useful for open data initiatives. However, usefulness for large GI datasets (like remote sensing) not yet fully clear ‚Äî huge data volumes are problematic.

5.2 Visualization of Geoinformation ‚Äî WMS & OGC API Maps

Web Map Service ¬∑ WMTS ¬∑ Styled Layer Descriptor ¬∑ OGC API Maps/Tiles/Styles

Requirements for Visualization Services

From INSPIRE View Services requirements:

  • Digital cartographic visualization of vector and raster data
  • Access via the internet, easy combination of distributed services
  • Access to further meta- and attribute data of visualized objects
  • No advanced requirements for the client ‚Äî thin client ideal; visualization directly in browser
  • Main part of rendering happens at the server side

Web Map Service (WMS)

WMS: First version ~2000. Current: WMS 1.3.0 (OGC 06-042 from 2006; also ISO 19128). A WMS returns rendered images (not raw data). It is a stateless service using HTTP GET and POST.

Two Types of WMS

Basic WMS

Provides only a cartographic visualization of geodata. Operations: GetCapabilities + GetMap.

Queryable WMS

Also allows querying information about visualized features. Operations: GetCapabilities + GetMap + GetFeatureInfo.

A WMS follows a layer-based approach. Displayed geodata can be 2D or 3D, static or time-variant. Cascading WMS link other WMS and combine their layers with their own.

WMS Operations

GetCapabilities
Mandatory. Returns XML service metadata: name/provider/abstract/keywords/fees, supported operations and formats, offered layers (with hierarchy, styles, CRS, bounding box, scale range, transparency, queryable flag).
GetMap
Mandatory. Returns a map image (e.g. PNG). Request parameters: SERVICE, VERSION, REQUEST, LAYERS, STYLES, CRS, BBOX, WIDTH, HEIGHT, FORMAT, TRANSPARENT. Key: no strict order of KVPs required.
GetFeatureInfo
Optional (queryable WMS only). Queries information about features at a specific pixel location in the map. Response is not well-defined ‚Äî can be HTML, plain text, or XML.

WMS Coordinate Systems

Map CS (Pixel Space)

  • Integer positive pixel coordinates
  • Origin (0,0) at upper-left corner
  • i = horizontal axis (west‚Üíeast), j = vertical axis (north‚Üísouth)
  • WIDTH/HEIGHT define map size in pixels

Layer CRS (Geographic Space)

  • WMS must support at least one CRS
  • For interoperability: should always support WGS84 geographic coordinates
  • Layers from different WMS can only be overlaid if they share a common CRS
  • CRS codes: EPSG (e.g. EPSG:4326 = WGS84 lat/lon), CRS:84 (WGS84 lon/lat), AUTO2 (custom)
  • WMS transforms from Layer CRS to Map CS
EPSG:4326 vs CRS:84 trap! EPSG:4326 = WGS84 with axis order latitude, longitude (y,x). CRS:84 = WGS84 with axis order longitude, latitude (x,y). This axis swap causes common interoperability bugs in WMS 1.3.0!

WMS Graphic Formats

  • Raster: GIF, PNG, JPEG, TIFF ‚Äî for street networks, use PNG (supports transparency)
  • Vector: SVG, WebCGM
  • A WMS must support at least one format with transparency that requires no browser plug-ins ‚Üí mostly PNG
  • TRANSPARENT=TRUE is crucial for overlaying distributed WMS layers!

WMS Profiles

GDI-DE Profile (WMS-DE_1.0): Specifies required CRS (EPSG:25832, EPSG:4258, etc.), language support, mandatory operations, layer naming conventions, legend URL, GetFeatureInfo response format.

INSPIRE View Service Profile: Implements INSPIRE Implementing Rule (EC Regulation 976/2009).

INSPIRE Implementing Rule vs. Technical Guidance (reminder): Implementing Rule = legally binding, defines WHAT, technology-independent, long-living. Technical Guidance = informative only, defines HOW, specifies technology, can be updated as technology changes. INSPIRE Technical Guidance for View Services defines two profiles: (1) ISO 19128 (WMS 1.3.0) and (2) Web Map Tile Service 1.0.0.

INSPIRE View Service QoS (legally binding!):

  • Performance: For a 470 KB image (800√ó600 px, 8-bit), initial response ‚â§ 5 seconds (at 90% of time = "normal situations")
  • Capacity: 20 parallel queries per second
  • Availability: 99% of time (= max. ~4 days per year downtime)

Web Map Tile Service (WMTS)

Instead of generating map images on-the-fly, WMTS pre-generates and caches tiles at fixed scales. This significantly improves performance for standard zoom levels but is less flexible for arbitrary areas/scales.

Use cases: background maps, web mapping (e.g., OpenStreetMap tile servers), mobile maps. OGC API ‚Äì Tiles is the modern ROA replacement.

Styled Layer Descriptor (SLD)

An extension to WMS that allows clients to define their own styling rules for layers. SLD uses Symbology Encoding (SE) to specify point symbols, line styles, polygon fills, text labels, etc. in XML.

5.3 Access to Geodata ‚Äî Encodings

GML ¬∑ GeoJSON ¬∑ GeoPackage ¬∑ Format Comparison

Requirements for Access Services (INSPIRE Download Services)

  • Access to raster and vector data in an open encoding
  • Support for spatial and thematic selection (filtering)
  • Possibly also writing new features (transactions)
  • Access via internet; easy combination of distributed access services

ESRI Shapefile (Legacy)

Well-established legacy format, but disadvantages for interoperability:

  • Not an open standard; consists of multiple files (.shp, .dbf, .shx, ...)
  • Attribute names max. 10 characters; strings max. 255 characters
  • Max. size 2 GB; no Unicode support

GML ‚Äî Geography Markup Language

GML: An XML dialect for encoding and transferring geodata. Provides both a machine-readable exchange format AND a mechanism for developing specific geodata schemata (application schemas) to support semantic interoperability.

GML Version History

  • GML 1.0 (~2000): Simple Feature Model only (vector geometries; no complex geometry/topology)
  • GML 2.0/2.1 (~2002): Extension for thematic attributes; XML Schema for data model; GML instance = XML file containing only geodata
  • GML 3.0/3.2.1/3.3 (~2005‚Äì2010): Full adoption of ISO 191xx standards; GML 3.2.1 = ISO 19136; added topology, time, observations, coverages, 3D features; rules for application schemas

Feature Model (ISO 19101)

A feature is a digital representation of a real-world object. A feature type is the class definition. A feature collection groups multiple features.

  • GML Profile: Subset of GML for specific use cases (reduces complexity)
  • GML Application Schema: Domain-specific extension of GML for specific data types (e.g., road network, buildings, land use). Described in XML Schema (XSD), often modeled in UML first (ISO 19109).
<gml:Point gml:id="p21" srsName="urn:ogc:def:crs:EPSG:6.6:4326"> <gml:pos dimension="2">45.67 88.56</gml:pos> </gml:Point>

Strengths: Highly flexible, extensible, supports complex modeling (topology, time, 3D). Weaknesses: Verbose files, slow to read/write, complex schemas.

GeoJSON

An IETF standard (RFC 7946, 2016) for encoding geographic data structures using JSON. Lightweight, readable, easy for web, compatible with many applications. Coordinate order in GeoJSON is always longitude, latitude (x,y) in WGS84.

{ "type": "Feature", "geometry": { "type": "Point", "coordinates": [-74.006, 40.714] }, "properties": { "name": "New York City" } }

Strengths: Readable, easy for web/APIs, compatible with JS/Leaflet/Mapbox. Weaknesses: Can get large for complex data, limited field types, no topology support.

GeoPackage

GeoPackage: OGC standard (version 1.4). A single SQLite database file containing both vector features AND raster data, plus metadata. Open, modern alternative to Shapefile.

Advantages: Up to 140 TB storage, single file, stores metadata for all layers, compressed (smaller than Shapefile), R-tree spatial indexing for fast queries, supports SQL queries, stores multiple geometry types in one file.

Disadvantages: Not all older GIS support it; SQLite complexity; not optimized for concurrent multi-user editing (file-based, not server-based).

5.4 Access to Feature-based Geodata ‚Äî WFS & OGC API Features

Web Feature Service ¬∑ Filter Encoding ¬∑ OGC API Features (WFS 3.0)

Web Feature Service (WFS)

WFS: First version ~2001. Current: WFS 2.0 (2009/10) = ISO 19142. Unlike WMS (returns images), WFS returns raw vector data (GML). Stateless service using HTTP GET / POST / SOAP. Allows interoperable read/write access to distributed geodatabases.

Four Types of WFS

Simple WFS

Read-only access using pre-defined queries. Operations: GetCapabilities + DescribeFeatureType + GetFeature.

Basic WFS

Read-only with user-defined (ad hoc) queries. All Simple WFS ops + GetPropertyValue.

Transactional WFS (WFS-T)

Reading and writing geodata (insert/update/delete). Basic WFS + Transaction operation.

Locking WFS

Supports locking features during transactions (simulates session on stateless protocol). WFS-T + LockFeature operation.

WFS Operations

GetCapabilities
Always mandatory. Returns: WFS instance name/provider/abstract, supported operations, offered feature types (names, bounding box, CRS).
DescribeFeatureType
Always mandatory. Returns GML Application Schema: which properties/attributes, data types, geometry types, relationships, CRS of the offered feature types. Needed before writing ad hoc queries.
GetFeature
Always mandatory. Returns a GML document with all features fulfilling the query. Queries can be ad hoc or stored. Uses Filter Encoding to formulate queries.
GetPropertyValue
Basic WFS and above. Returns the attribute values (not full features) fulfilling a query. Returns as key-value pairs.
Transaction
Transactional WFS. Supports insert, update, delete of features. New/updated features sent as GML objects. Can include LockID. Response confirms success/failure.
LockFeature
Locking WFS. Locks specific features so no other client can modify them. Releases lock after successful transaction. Workaround for enabling transactions over stateless protocol. Returns LockID.

Filter Encoding Standard (FES)

Filter Encoding: OGC standard (current: FES 2.0 / ISO 19143) for formulating queries on geo-objects as XML. Independent of specific query language, DBMS, or application. Supports both thematic queries (attribute conditions) and spatial queries (geometry relationships like Contains, Intersects, etc.).

Example: Find all lakes contained within Algonquin Park ‚Üí uses both spatial filter (Contains) and thematic filter (FeatureType = Lake).

WFS Conclusions

  • WFS allow interoperable access to distributed geodatabases in principle
  • Too many heterogeneous application schemata hamper interoperability in practice
  • Transactional WFS are rare (need careful security and data curation)
  • There are fewer WFS than WMS (because WFS gives full data access)
  • Performance issues with large datasets ‚Äî is web-based transfer of big geodatasets sensible?

INSPIRE Download Service (WFS Profile)

INSPIRE distinguishes two types: (1) Dataset Download ‚Äî pre-defined dataset retrieval, (2) Direct Access Download ‚Äî supports user-defined queries. Technical Guidance defines three profiles: Atom-feed (pre-defined), WFS (pre-defined), WFS + Filter Encoding (direct access).

QoS requirements: Capacity = 10 parallel requests/second (vs. 20 for WMS); Availability = 99%; Performance: GetMetadata = 10s; GetDataSet = 30s initial + sustained 0.5 MB/s or 500 objects/s.

OGC API ‚Äì Features (WFS 3.0)

The modern ROA/OpenAPI successor to WFS 2.0. Part 1: Core (2022), Part 2: CRS by Reference, Part 3: Filtering (2024) using CQL2 (Common Query Language 2.0).

URL pattern: {root}/collections/{collectionId}/items. Returns GeoJSON by default. Can also return GML. Direct browser access possible ‚Äî human-readable.

Key advantages over WFS 2.0: OpenAPI-based = self-documenting, discoverable; GeoJSON default = easier web integration; no complex SOAP/XML needed; general-purpose web search engines can index it; simpler for non-GIS developers.

5.5 Access to Coverage/Raster Geodata ‚Äî WCS & OGC API Coverages

Coverage Model ¬∑ Web Coverage Service ¬∑ OGC API Coverages

Coverage Model (ISO 19123)

Coverage: Supports mapping from a spatiotemporal domain to attribute values where attribute types are common to all geographic positions within the spatiotemporal domain. This is the field view of geographic data.

Discrete Coverage

Returns the same attribute value for every direct position within a single spatial object (finite domain). Example: classified land use raster ‚Äî all pixels in one class have the same value.

Continuous Coverage

Returns different values at different positions ‚Äî can be subdivided infinitely. Example: digital elevation model, temperature field. WCS focuses on discrete coverages (continuous not further specified).

A Rectified Grid is a grid for which there is an affine transformation between grid coordinates and coordinates of an external CRS (standard raster/image). Grid points represent the center of sample spaces (pixels). Interpolation functions (nearest neighbor, bilinear, etc.) handle value estimation between grid points.

Web Coverage Service (WCS)

WCS: First version ~2002. Current: WCS 2.0 (OGC 09-110r4, 2010) / WCS 2.1 (2018). Stateless service. Focuses on interoperable access to raster/coverage data (2D, 3D, time-variant, multi-band). Complements WFS (which focuses on vector features).

WCS Operations

GetCapabilities
Mandatory. Returns: WCS instance metadata, offered operations, offered coverages (abstract, CRS, formats).
DescribeCoverage
Mandatory. Returns detailed metadata for specific coverage(s): attributes, data types and domains, supported CRS, spatial/temporal extent, resolution, supported interpolation methods.
GetCoverage
Mandatory. Request parameters: CRS, spatial/temporal subset (Trimming = subset of coverage; Slicing = a dimension cut, e.g. 2D slice from 3D dataset), spatial resolution (resampling), which bands/layers, output format. Returns a coverage file (TIFF, NetCDF, HDF, ...) or a download URL.

GetCoverage Subset Types

Trimming

Subset along an axis to produce a coverage of the same dimensionality. E.g., spatial bounding box subset of a 2D raster = still a 2D raster.

Slicing

Subset that reduces dimensionality by one. E.g., select one time slice from a 3D (space √ó time) dataset = get a 2D spatial raster.

WCS Conclusion

  • Distributed WCS allow interoperable access to coverages ‚Äî mainly remote sensing and climate data
  • Fewer WCS implementations than WMS
  • Performance issues for large rasters ‚Äî smart aggregation, generalization, and filtering needed to reduce data transfer

OGC API ‚Äì Coverages

Website: ogcapi.ogc.org/coverages/. Part 1: Core available as Draft 0.0.7. Part 2: Filtering ‚Äî planned. Returns data encoded in CoverageJSON (OGC Community Standard, Aug 2023). Example at TU Dresden Klimakonform project: daily temperature data accessible via simple URL with bbox, datetime, and CRS parameters.

5.6 AAA ‚Äî Authoritative German Geoinformation

AFIS ¬∑ ALKIS ¬∑ ATKIS ¬∑ GeoInfoDok ¬∑ NAS Interface

The AAA System

AAA Model: Amtliches Festpunktinformationssystem / Amtliches Liegenschaftskatasterinformationssystem / Amtliches Topographisch-Kartographisches Informationssystem. Developed by AdV (Arbeitsgemeinschaft der Vermessungsverwaltungen der Bundesl√§nder). Based on ISO 19xxx standards and OGC specifications.

üîµ AFIS

Amtliches Festpunktinformationssystem ‚Äî Authoritative Reference Point Information System. Geodetic control points.

üü° ALKIS

Amtliches Liegenschaftskatasterinformationssystem ‚Äî Authoritative Real Estate Cadastre Information System. Property/parcel data (replaces ALB + ALK).

üü¢ ATKIS

Amtliches Topographisch-Kartographisches Informationssystem ‚Äî Authoritative Topographic-Cartographic Information System. Topographic/landscape data (replaces old ATKIS).

AAA has been (almost) fully implemented in all German federal states since ~2015. The model is documented in GeoInfoDok (Dokumentation zur Modellierung der Geoinformationen des amtlichen Vermessungswesens).

Historical Development

Before AAA (~1995): separate, technology-specific systems (ALB, ALK, old ATKIS) using AdV-specific technical foundations. With AAA (~2015): unified model based on ISO 19100 standards, OGC specs, and W3C recommendations ‚Äî fully interoperable via WFS/GML.

NAS ‚Äî Normbasierte Austauschschnittstelle

The NAS (standards-based exchange interface) is the interface for communication between different vendor-specific AAA software systems. Key requirements (~2002):

  • Common description of objects, operations, and information products
  • Automatic derivation of schemata from UML models
  • XML-based, using market standards (ISO 19xxx, OGC WFS/GML)

AAA Architecture

The AAA model is structured in layers: ISO 19100 Series (foundation) ‚Üí AFIS-ALKIS-ATKIS-Profiles ‚Üí AFIS-ALKIS-ATKIS-Specifications ‚Üí AAA Application Schema + Metadata + Registries ‚Üí Operations (WFS interface) ‚Üí Presentation (symbols, rules).

AAA Operations: Locking, Unlocking, Updating, Inserting, Using. Used by: (1) Authorities/surveying agencies for data acquisition and maintenance, (2) Secondary databases, (3) External users (energy providers, etc.) for read access.

URNs in AAA

AAA uses URNs (Uniform Resource Names) to uniquely identify CRS, units of measurement (UoM), and object IDs:

<gml:Point srsName="urn:adv:crs:DE_DHDN_3GK3"> <gml:pos>3534880.00 5395220.00</gml:pos> </gml:Point> <amtlicheFlaeche uom="urn:adv:uom:m2">626</amtlicheFlaeche> <istGebucht xlink:href="urn:adv:oid:DEHE123400229002"/>

AAA Products

Products and presentations are processed base data sets, ideally created "on-the-fly" and "on demand". Some products are stored separately (require additional symbol/styling rule sets). Products can be ATKIS DLM topographic data, cadastral maps, DGMs, etc.

AAA ‚Äî INSPIRE Relationship

Study (Reinwarth 2010): ALKIS cadastral parcels CAN be transformed into INSPIRE Cadastral Parcels using FME (Feature Manipulation Engine). This is still "an easy case" ‚Äî the AAA ‚Üí INSPIRE transformation is generally complex and ongoing research topic.

Current development: GeoInfoDok version 7 (in progress) will update the AAA model to also include 3D city models and align with INSPIRE Metadata Rules.

5.7 INSPIRE Harmonized Data Specifications

Development Process ¬∑ Implementing Rules ¬∑ Application Schemas

Development Approach

Drafting INSPIRE Implementing Rules for harmonized geodata specs followed two steps:

  1. Develop a conceptual model and methodology for the specifications ‚Äî based on ISO 19100 Standards
  2. Develop the data schemata for each Annex topic ‚Äî done in Thematic Working Groups (TWGs)

Specs follow a hierarchy: Framework for harmonized data specification (DT DS ‚Äî common elements all topics share) ‚Üí Thematic Data Specifications (per TWG, per Annex topic).

  • Implementing Rules for Annex I & II: updated version 2014
  • Implementing Rules for Annex III: first version 2013

What INSPIRE Data Specifications Contain

  • Based on ISO 191xx standards
  • UML diagrams showing harmonized geodata models (object types/classes, geometry presentations, attributes, which are mandatory)
  • Identifiers, relationships, core attributes, temporal dimension, versioning
  • Reference to INSPIRE network services for delivery

Current Challenges and Research Questions

  • Are these specifications implementable and consistent?
  • Do we need less complex specifications?
  • How to transform/migrate existing databases into INSPIRE data (ad-hoc)? Who, when, which data?
  • Can INSPIRE data also be used in national applications ‚Äî what is missing?
  • How to converge ALKIS/ATKIS and INSPIRE schemas?

Alternative Encodings in INSPIRE (Post-REFIT)

Following the 2016 REFIT evaluation that stepped back from full harmonization requirements, INSPIRE now also accepts "Good Practices" for alternative encodings:

  • GeoJSON encoding of INSPIRE datasets ‚Äî for simpler web-based access
  • GeoPackage encoding of INSPIRE datasets ‚Äî for more complex data including raster

This makes INSPIRE data more accessible and reduces implementation burden compared to strict GML/WFS requirements.
